# Livvie Code Review v2 — Custom Agent Files Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Replace hardcoded perspectives with user-defined agent `.md` files in `.github/livvie_code_review_agents/`.

**Architecture:** Each `.md` file = 1 review agent (YAML frontmatter + markdown body as system prompt). The action scans the folder via GitHub API, parses each file, and runs all agents through the existing pipeline. The `Perspective` interface remains the internal contract — agent-loader converts `.md` files into `Perspective[]`.

**Tech Stack:** TypeScript, node24, Vitest, @actions/core, @octokit/rest. No new dependencies (custom regex frontmatter parser).

**Breaking change:** Yes. v2 requires agent files. The `perspectives` input is removed. v1 stays frozen at current tag.

---

## Decisions Log

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Empty agent folder | **Fail-fast** | User asked for breaking change. v2 is explicit — no silent defaults. |
| `perspectives` input | **Removed** (not deprecated) | Clean break. v1 users stay on v1 tag. |
| `review-instructions-file` | **Kept** | Orthogonal — defines project context, not agent personas. |
| Frontmatter parser | **Custom regex** | 6 scalar fields, ~30 lines. gray-matter = 70+ transitive deps. |
| `SHARED_REVIEW_RULES` | **Auto-appended** by action | Output format contract is an implementation detail. Users write persona only. |
| Agent ID source | **Filename stem** | `security.md` → id `security`. Stable, deterministic. |
| `id` frontmatter field | **Not supported** | YAGNI. Filename is the ID. |
| `focus` frontmatter field | **Not supported** | Derived from description or first heading in body. |
| Duplicate agent names | **Fail-fast** | Names used as `foundBy` keys. Ambiguity = bug. |

## Agent File Schema

### Location
```
.github/livvie_code_review_agents/
  security.md
  performance.md
  generalist.md
  my-custom-agent.md
```

### Format
```markdown
---
name: Security Reviewer
description: Reviews code for security vulnerabilities and risks
enabled: true
model: null
temperature: 0.1
---

You are a **Security Reviewer**. You review code for security vulnerabilities.

## Your focus areas
- Injection: SQL injection, command injection, XSS
- Secrets: hardcoded API keys, tokens, passwords
...
```

### Frontmatter Fields

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `name` | string | **yes** | — | Human-readable name shown in PR comments. Must be unique. |
| `description` | string | no | `""` | Short description (logged at startup). |
| `enabled` | boolean | no | `true` | If `false`, agent is skipped. |
| `model` | string | no | `null` | Override global model for this agent. |
| `temperature` | number | no | `0.1` | LLM temperature override. |

### Body (System Prompt)
Everything after closing `---`. The action appends `SHARED_REVIEW_RULES` automatically — users do NOT include format rules.

### No Frontmatter Fallback
If file has no `---` delimiters, entire file is the body. Name derived from filename (`security.md` → name `"security"`). Enables ultra-minimal agent files.

## File-by-File Changes

### NEW: `src/agent-loader.ts`

**Responsibilities:**
1. Discover `.md` files via `octokit.rest.repos.getContent({ path: agentsDir, ref })`
2. Parse YAML frontmatter + body (custom regex, ~30 lines)
3. Validate: name required, no duplicates, body non-empty, enabled check
4. Convert to `Perspective[]` (id from filename stem, systemPrompt = body + SHARED_REVIEW_RULES)
5. Log each loaded agent via `core.info()`

**Exported functions:**
```typescript
export async function loadAgents(
  octokit: Octokit,
  owner: string,
  repo: string,
  ref: string,
  agentsDir: string,
): Promise<{ perspectives: Perspective[]; agentModelOverrides: Map<string, { model: string | null; temperature: number }> }>
```

**Validation rules:**

| Condition | Behavior |
|-----------|----------|
| `name` missing or empty | **Fail-fast** naming the file |
| `name` duplicates another agent | **Fail-fast** listing both files |
| `enabled: false` | Skip silently (debug log) |
| Body empty | **Fail-fast** |
| No frontmatter | Treat entire file as body, derive name from filename |
| `temperature` outside 0-2 | Warn, clamp to 0.1 |
| Folder doesn't exist | **Fail-fast**: "No agent files found at {path}. Create at least one .md file." |
| Folder empty | Same as above |
| All agents disabled | **Fail-fast**: "All agents are disabled." |
| > 10 agents loaded | Log warning about LLM call volume |

**Stage 3 (Review Matrix):** Add model/temperature override merging:
```typescript
const matrixCalls: Array<{
  batch: Batch;
  perspective: Perspective;
  modelOverride?: string;
  temperatureOverride?: number;
}> = [];

for (const batch of batches) {
  for (const perspective of perspectives) {
    const overrides = config.agentModelOverrides.get(perspective.id);
    matrixCalls.push({
      batch,
      perspective,
      modelOverride: overrides?.model ?? undefined,
      temperatureOverride: overrides?.temperature ?? undefined,
    });
  }
}
```

## Example Agent Files (for README/docs)

### `.github/livvie_code_review_agents/generalist.md`
```markdown
---
name: General Reviewer
---

You are a **General Code Reviewer**. You review code for issues that span multiple concerns.

## Your focus areas
- Edge cases: null handling, boundary conditions, race conditions
- Correctness: logic errors, wrong variable references
- Documentation: missing doc comments for public APIs
- Consistency: inconsistent error handling within the same module
- Testing: obviously untested code paths

Flag anything that a thorough senior developer would notice.
```

### `.github/livvie_code_review_agents/security.md`
```markdown
---
name: Security Reviewer
---

You are a **Security Reviewer**. You review code for security vulnerabilities.

## Your focus areas
- Injection: SQL injection, command injection, XSS, path traversal
- Secrets: hardcoded API keys, tokens, passwords
- Authentication: missing auth checks, privilege escalation
- Input validation: missing sanitization, unsafe deserialization
- Data exposure: sensitive data in logs or error messages

Only flag genuine security risks.
```

*(Similar for performance.md, architecture.md, code-quality.md)*
